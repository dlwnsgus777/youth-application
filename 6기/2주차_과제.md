## 토스(Toss) 비전 · 일하는 방식 · 서비스 · 조직도 분석

### 1. 토스는 어떤 비전을 가지고 있나요?

#### 핵심 비전: **“새로운 차원의 금융(New Dimension of Finance)”**
- 기존 금융의 불편함을 다시 설계해 누구나 쉽게 사용하는 금융 서비스를 만든다.
- 금융 경험을 단순하고 상식적이며 접근하기 쉬운 구조로 재해석한다.
- 유저가 아무것도 몰라도 이해할 수 있는 UX를 지향한다.

> 이승건 대표는 “금융은 원래 어렵지 않습니다. 어렵게 만든 것은 금융 산업의 관행이었고, 우리는 그 상식을 새로 쓰고 싶습니다.”라고 강조한다.

#### 생각 메모
- 토스증권을 이용하며 초보자가 진짜 사용하기 편하다는 인상을 받음.
- “이제 한번만 더 인증하면 돼요”와 같은 친절한 안내 문구가 가장 기억에 남음.

#### 핵심 가치(Core Values)
1. **Mission over Individual — 개인보다 팀의 미션을 우선한다**  
   - 설명: 개인 성과보다 팀의 임팩트 창출이 우선순위다.  
   - 엔지니어 관점: 서비스·도메인을 가리지 않고 팀 성공을 위해 필요한 일을 맡는다.
2. **Aim Higher — 더 높은 수준을 추구한다 (구 Go the Extra-mile)**  
   - 설명: ‘좋음’이 아닌 ‘탁월함(Excellent)’을 기준으로 끊임없이 개선한다.  
   - 엔지니어 관점: 코드 품질, 장애 대응, 안정성에서 높은 기준을 고집한다.
3. **Focus on Impact — 해야 할 일 중 임팩트가 큰 한 가지에 집중한다**  
   - 설명: 바쁨이 아니라 결과와 영향으로 가치를 판단한다.  
   - 엔지니어 관점: 가장 큰 병목과 중요한 문제를 찾아 해결하는 능력을 중시한다.
4. **Question Every Assumption — 모든 가정을 의심하고 본질을 본다**  
   - 설명: “원래 이렇게 해왔어요”라는 답을 용납하지 않고 전제를 검증한다.  
   - 엔지니어 관점: 데이터·로그 기반으로 구조적 문제를 분석하고 기술 부채를 제거한다.
5. **Execution over Perfection — 완벽보다 실행을 우선한다**  
   - 설명: 완벽한 설계보다 빠른 실행과 학습 루프(MVP → 반응 → 개선)를 중시한다.  
   - 엔지니어 관점: 작게 만들고 빠르게 배포하면서 실제 유저 데이터로 판단한다.
6. **Learn Proactively — 주도적으로 학습한다**  
   - 설명: 스스로 학습 계획을 세우고 필요 시 누구의 지시 없이도 역량을 확장한다.  
   - 엔지니어 관점: 문서가 없어도 코드를 파고들며 도메인과 스택을 스스로 익힌다.
7. **Move with Urgency — 신속하게 움직인다**  
   - 설명: 문제 해결, 의사결정, 실험에서 속도감을 경쟁력으로 본다.  
   - 엔지니어 관점: 장애 대응과 롤백을 빠르게 수행하고 실험적 배포를 반복한다.
8. **Ask for Feedback — 피드백을 적극적으로 구한다 (구 Radical Candor)**  
   - 설명: 좋은 피드백 문화는 요청하는 사람에서 시작된다고 본다.  
   - 엔지니어 관점: 코드 리뷰와 설계 검토를 자발적으로 요청하며 근거 기반 토론을 즐긴다.

#### 생각 메모
- 실제로 토스의 컨퍼런스 내용이나 인터뷰 등을 참고해볼 때 핵심가치를 매우 잘 지키는 조직이라는 생각이 들었음
- 토스의 핵심가치는 내가 추구하는 개발자의 모습과 일치하는 부분이 많아서 토스에 합류했을 때 많은 걸 얻어갈 수 있겠다라는 생각이 들었음

## 컬리는 어떤 방식으로 일하나요?
*(토스 공식 홈페이지, Oopy 조직문화 문서, Careers 인터뷰, 토스 유튜브 등을 기반으로 정리)*

### 1) Tribe → Squad 구조 (제품 단위 독립 조직)

- 토스는 제품 중심(Product-led) 조직을 운영하며, 제품별로 **Tribe → Squad** 구조를 가지고 있다.
  - 하나의 스쿼드는 **PM, 개발자, 디자이너, 데이터 분석가**가 포함된 작은 독립 팀이다.
  - 스쿼드는 **문제 정의, 목표 설정, 기획, 개발, 배포**까지 대부분의 의사결정을 자체적으로 수행한다.

### 2) Maker 중심 (현장 주도 문제 해결)

- 토스는 PM·디자이너·엔지니어를 모두 **Maker(메이커)**로 부르며, 문제 해결의 주체로 인정한다.
  - 문제 정의(Problem Framing) → 해결책 설계 → 실험까지 메이커가 직접 문서로 작성하고 실행한다.
  - 경영진이나 외부 조직에서 요구사항을 전달하는 방식이 아니라, **스쿼드가 현장에서 문제를 발견하고 해결**한다.

### 3) 문서 기반 의사결정 (Writing → Discussion → Execution)

- 토스는 빠른 말보다 **정제된 문서**를 통해 생각을 공유하고 결정한다.
- 문서는 보통 다음 내용을 포함한다:
    - 해결하려는 문제의 정의
    - 목표 지표(Impact Metric)
    - 가설과 해결 방안
    - 예상 리스크 및 단계별 실행 계획
- 회의는 문서를 기반으로 리뷰하는 방식이며, 문서의 논리적 완성도가 의사결정의 핵심 기준이다.

### 4) 높은 투명성과 직접적 피드백 (Radical Transparency)

- 토스는 핵심가치 Ask for Feedback을 통해 **직접적이고 구체적인 피드백 문화를 공식적으로 명시**하고 있다.
- 장애 리포트, 실험 결과, 지표 데이터 등 중요한 정보가 팀 전체에 투명하게 공유된다.
- 의견을 돌려 말하기보다 **팩트를 기반으로 명확하게 전달**하는 방식이 기본이다.
- 토스 유튜브와 인터뷰에서 반복적으로 검증된 문화적 특징이다.

### 5) Think Slow → Execute Fast (깊은 사고 + 빠른 실행)

- 공식 Core Value 3.1 중 *Execution over Perfection*의 정신과 연결된다.
- 문제 정의와 가설 설정은 깊게 진행하지만,  
  **실행과 실험은 빠르게 반복**하여 실제 데이터를 기반으로 개선한다.
- 토스의 다양한 제품(토스증권/토스뱅크 등)이 초기 출시 후 빠른 업데이트를 반복한 사례가 이를 증명한다.

### 6) 하이 퍼포먼스 문화 (명확한 기대치 + 큰 자율성)

- 핵심가치 "Aim Higher", "Focus on Impact"를 바탕으로 **높은 기준과 명확한 책임**을 강조한다.
- 자율성은 크지만, 임팩트가 낮은 활동은 성과로 인정되지 않는다.
- 분기 단위 성과 리뷰가 존재하며, 이는 여러 인터뷰에서 사실로 확인되었다.
- High performer 중심의 조직이라는 점도 공식적으로 반복 언급되는 특징이다.

### 7) 실험 중심 데이터 문화 (A/B Test & Data Driven)

- 스쿼드에는 데이터 분석가가 포함되어 있으며, 의사결정 단계마다 **데이터가 근거**가 된다.
- 새로운 기능은 A/B 테스트 또는 실험 형태로 출시되며, 고객 행동 데이터로 성공 여부를 판단한다.
- “추측이 아니라 측정”이라는 원칙은 토스의 제품 팀 인터뷰에서 공개적으로 언급되었다.

### 8) 개발자 중심 운영(Developer Ownership)

- 개발자는 구현뿐 아니라 **설계, 배포, 모니터링, 장애 대응까지 end-to-end 책임**을 가진다.
- 토스 테크 블로그에 따르면 RCA(장애 후 재발 방지 문서) 작성 및 운영 개선 또한 개발자의 역할이다.
- DevOps 성숙도가 높은 편이며, 스쿼드의 실행 속도를 높이는 기반이 된다.

### 9) 작은 팀, 빠른 의사결정 (Small Teams, Fast Decisions)

- 스쿼드는 보통 4~7명 정도의 소규모 팀으로 운영된다.
- 구성원이 적어 이해관계가 단순하며, 의사결정 속도가 매우 빠르다.
- 토스 인터뷰와 문화 소개 자료에서 “작은 팀이 더 빠르다”는 철학이 반복적으로 언급된다.

#### 생각 메모
- 제품을 발전시키기 위해 끊임없이 동료들과 토근하고 개발하고 검증하는 단계를 반복하는 것 같음
- 예전 회사에서 특별조직이라고 해서 TF형식으로 각 팀에서 팀원들을 한명씩 뽑아서 TF 조직을 구성하고 특정 목적을 달성하기 위해 일하는 방식이 있었는데 
  - 토스에서는 이걸 더 심층적으로 하는 것 같음
  - 토스에서는 스쿼드의 비즈니스 목적을 더 뚜렷하게 하여 성과를 유도하는 것 같음
    - 핵심가치에서 계속 언급되는 속도, 검증이 더 시너지가 날 수 있는 방식인 것 같음

## 토스는 어떤 서비스를 운영하고 있나요?

---

### 1. 비바리퍼블리카(본사, toss core – 토스 앱)

#### **주력 서비스**
- 송금·이체
- 계좌/카드 조회, 자산관리(PFM)
- 신용 조회 및 점수 관리
- 청구서 납부·세금 납부
- 대출 비교, 금융상품 추천
- 인증·보안 인프라, 홈 화면 개인화

#### **비교 대상**
- 카카오페이
- 네이버페이
- 페이코
- 뱅크샐러드

#### **차별점**
- **금융 전 범위를 아우르는 올인원 슈퍼앱**
- **원탭 송금·한 화면 자산 보기 등 금융 UX 최적화**
- **금융 데이터 기반 개인화 추천**
- 앱 하나에서 송금 → 신용 → 투자 → 보험 → 세금까지 처리 가능

### 2. 토스뱅크(toss bank)

#### **주력 서비스**
- 예금·적금
- 신용대출·마이너스 통장
- 연체/채권관리
- ML 기반 심사 모델
- 리스크 관리
- 실시간 계정계(Core Banking System)

#### **비교 대상**
- 카카오뱅크
- 케이뱅크
- 전통 시중은행 앱

#### **차별점**
- **실시간 처리 중심의 자체 계정계 개발**
- 복잡한 은행 업무를 모바일 UX 기준으로 재설계
- 중·저신용 고객 중심 대출 전략
- ML 기반 고도화된 여신 심사 모델

### 3. 토스증권(toss invest)

### **주력 서비스**
- 국내/해외 주식 거래
- ETF·채권 투자
- 자동 투자(AI 포트폴리오)
- 주문·체결 시스템
- 투자 콘텐츠, 종목 분석

### **비교 대상**
- 키움증권
- 미래에셋 MTS
- 신한알파, 나무 등 모바일 증권사

### **차별점**
- **초보자 중심의 직관적 MTS UX**
- 해외주식 환전 자동화 및 매매 절차 단순화
- 콘텐츠 기반 투자 정보 제공
- 실시간 스트리밍 기반 저지연 거래 시스템

### 4. 토스보험(toss insurance)

### **주력 서비스**
- 보험 비교·가입
- 기존 보험 분석 및 최적 보장 추천
- OCR 기반 보험금 청구 자동화
- 약관 분석 AI

### **비교 대상**
- 보맵
- 카카오페이 보험
- 각 보험사 자사 앱

### **차별점**
- **약관 분석, 증빙 OCR 등 AI 기반 자동화 수준이 높음**
- 보험금 청구 과정의 간소화·자동화
- 금융데이터 기반 초개인화 보험 추천 모델
- 보험 가입/관리 UX가 매우 단순함

### 5. 토스페이먼츠(toss payments)

### **주력 서비스**
- 온라인 결제(PG)
- 카드·계좌·간편결제 연동
- 정산·입금 관리
- Fraud Detection(부정거래 탐지)
- 결제 API/SDK 제공

### **비교 대상**
- KG이니시스
- 나이스페이
- 다날
- 네이버페이·카카오페이 결제 모듈

### **차별점**
- **개발자 친화적인 PG (문서/SDK/연동 난이도 낮음)**
- 일/익일 정산 등 빠른 정산 사이클
- 실시간 부정거래 탐지 시스템
- 토스 슈퍼앱과 자연스러운 간편결제 연동

### 6. 토스플레이스(toss place)

### **주력 서비스**
- POS(Point of Sale) 시스템
- QR·카드 결제 단말 연동
- 매출/재고/직원 관리
- 오프라인 매장 운영 SaaS
- 매출 인사이트 제공

### **비교 대상**
- 도도포인트
- 스마트로/다날 POS 단말
- 다양한 POS SaaS 기업

### **차별점**
- **POS + 결제 + 매출 분석 + 금융 서비스를 통합 제공**
- 금융 데이터를 기반으로 대출·정산 등 연결 가능
- 오프라인 결제 UX와 속도 개선
- 하나의 플랫폼에서 매장 운영 전반을 처리

### 7. 토스인컴(toss income)

### **주력 서비스**
- 세금 신고 자동화(부가세·종소세)
- 소득 인증
- 사업자 금융 서비스
- 프리랜서 계약/정산 도구

### **비교 대상**
- 홈택스, 손택스
- 삼쩜삼
- 사업자 매출 관리 앱(NICE 지니데이 등)

### **차별점**
- **금융데이터 기반 신고 자동화**(거래·매출·계좌 데이터 활용)

## 컬리는 어떤 비전을 가지고 있나요?

컬리는 “프리미엄 커머스”를 표방하며, 단순한 식품 배송 플랫폼이 아닌

### 최상위 비전(WHY)

#### 더 나은 일상을 제안하는 프리미엄 커머스
- 단순 상품 판매가 아니라 고객의 생활 수준을 높이는 커머스 경험을 제공하는 것이 목표.

  “컬리는 품질이 우선이며, 가장 높은 기준을 지켜야 한다.”

- 기존 유통사가 시도하지 않은 물류 혁신을 직접 수행하는 구조.

#### 신뢰 기반의 큐레이션 커머스 모델

## 컬리는 어떤 방식으로 일하나요?

### 1. 문서 중심(Documentation-first) 소통 문화

컬리는 말보다 **문서 기반 의사소통**을 매우 선호한다.

- Notion / Confluence / GitHub Wiki 등 문서 작성이 자연스럽게 문화로 자리잡음
- 설계 변경, 서비스 정책, 이슈 대응 결과를 문서로 기록
- 조직 간·도메인 간 지식 공유를 위해 블로그까지 적극 활용
- 신규 입사자 온보딩이 빠른 이유도 문서화 문화 덕분

문서는 단순 기록이 아니라 **의사결정의 근거**이자 팀의 학습자산으로 사용된다.

### 2. 데이터 기반(Data-driven) 의사결정 및 커뮤니케이션

컬리 개발팀의 소통 방식은 "감"이 아니라 **데이터로 설득**하는 구조이다.

- A/B 테스트 → 기능 개선 판단
- 로그/알람 지표 → 장애 원인 분석
- 수요 예측 모델 결과 → 운영 계획
- 피킹 알고리즘 개선 → 시뮬레이션 수치로 검증
- UI/UX 변경 → 전환률·재구매율 기반으로 평가

모든 변경과 주장에 **지표가 근거로 따라붙는** 형태가 기본 패턴이다.

### 3. 크로스펑셔널 협업(Cross-functional Collaboration)

컬리는 하나의 기능이 여러 조직을 동시에 건드리는 커머스+물류 도메인 특성상,
개발팀은 아래와 같은 팀들과 매일 자연스럽게 협업한다.

- MD팀
- 물류센터 운영팀
- 재고/SCM 팀
- CX(고객센터)팀
- 데이터/머신러닝 팀
- 플랫폼/인프라 팀

특히 물류 시스템(WMS, TMS)은 **실제 현장 직원과 공동 논의**하며 설계해야 하므로  
개발자는 현장 상황까지 고려한 문제 해결 능력이 필요하다.

### 4. 빠르고 직접적인 피드백 문화

컬리는 완곡한 표현보다 **즉각적이고 직설적인 커뮤니케이션**을 한다.

- 문제 발견 → 즉시 공유
- 장애 발생 → 비난보다 빠른 복구 우선
- 결정 지연을 싫어하고 빠른 합의·빠른 실행 선호
- 리뷰, 코드 논의도 명확한 근거 기반으로 직설적 피드백

이는 커머스·물류에서 **속도가 고객 경험 품질**과 직결되기 때문.

### 5. RFC + 코드리뷰 중심의 기술 소통

큰 변경 시에는 RFC(Request For Comment)를 작성해 팀 전체의 피드백을 요청한다.

- 아키텍처 변경
- 신규 서비스 설계
- 주요 기능 고도화
- 장애 재발 방지 제안

코드리뷰는 필수이며, 단순 문법 지적이 아니라  
“문제 정의 → 해결 방식 → 운영 영향”까지 검토하는 방식으로 진행된다.

### 6. 실험 기반(Experiment-driven) 커뮤니케이션

컬리 개발팀의 전형적인 논의 방식:

1) 문제 정의
2) 가설 설정
3) 실험 설계
4) 지표 기반 결과 분석
5) 개선

즉, 소통 자체가 **가설 기반 실험 프로세스**로 이루어진다.

### 7. DevOps & 운영 중심 업무 방식

컬리는 운영을 아주 중요하게 보기 때문에,
개발자가 서비스의 전체 라이프사이클을 직접 책임진다.

- 배포 → 모니터링 → 장애 대응 → 원인 분석(RCA) → 개선
- 로그/알람 정책을 개발자가 직접 설계
- 고객 영향이 큰 장애는 우선순위 최상
- 운영 자동화 스크립트/툴도 직접 구축

이는 기술블로그에서 가장 자주 강조되는 컬리 개발팀의 핵심 문화다.

### 8. 기술 공유(Tech Blog) 기반 학습 문화

컬리는 외부 기술블로그 운영을 내부 학습과 문화의 일부로 활용한다.

- 문제 해결 과정 공유
- 설계 변화 이유 기록
- 물류·커머스 도메인 지식 축적
- 신규 입사자의 빠른 온보딩
- 팀 간 학습 격차 감소

기술 공유는 개발자 성장뿐 아니라  
컬리 조직 전체의 **지식 축적 시스템**으로 기능한다.

### 9. 고객 중심 가치 기반 협업

최종 의사결정 기준은 "고객 영향"이다.

- 배송 지연 → 고객 경험 악화
- 재고 오류 → 오배송/환불 발생
- 후기 이미지 품질 → 구매 전환율에 영향
- 검색 품질 → 고객 탐색 경험 좌우

따라서 기술적 논의도 결국  
**“이 변경이 고객에게 더 좋은 경험인가?”**를 기준으로 결론을 낸다.

#### 생각메모
- 토스와 상당히 비슷함.
  - 토스는 새로운 도전, 변화를 말하지만 컬리에서는 도메인 성숙도를 더 중요하게 생각하는 것 같다는 느낌을 받았다.
